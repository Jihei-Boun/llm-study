# 07. RAG

<br>

## 한 줄 설명

RAG = AI가 답하기 **전에**, 관련 자료를 **찾아 읽고**(Retrieval), 그 내용을 **참고해서**(Augmented) **답을 만드는**(Generation) 방식

<br>

## 이전 노트와의 연결

06번 **Inference & Generation**까지 배웠다면, AI는 이렇게 동작함:

```text
질문 → Transformer → 확률표 → 토큰 선택 → 반복 → 답변 완성
```

여기서 AI는 **자기 머릿속(학습 때 본 데이터)**만으로 답함.

**RAG**는 그 앞에 한 단계가 더 붙음:

```text
질문 → (관련 자료 검색) → 질문 + 자료를 함께 넣음 → Transformer → 답변
```

비유:

| 단계 | 06번 (일반 AI) | 07번 (RAG) |
|------|----------------|------------|
| 시험 | **암기한 것만**으로 답 | **교과서를 펼쳐 보고** 답 |
| 요리 | 레시피를 **외워서** 요리 | **냉장고를 열어** 재료 확인 후 요리 |

<br>

✅ 정리

> Inference & Generation이 **"머릿속만으로 글 쓰기"**라면,  
> RAG는 **"찾아 읽고 나서 글 쓰기"**

<br>

## RAG 이름 뜻 풀기

| 영어 | 뜻 | 쉬운 말 |
|------|-----|---------|
| **R**etrieval | 검색·찾기 | 도서관에서 **관련 책 찾기** |
| **A**ugmented | 보강·추가 | 찾은 내용을 **질문 옆에 붙이기** |
| **G**eneration | 생성 | 그걸 보고 **답 작성** |

합치면: **"찾아서 붙인 뒤 생성"**

<br>

## 왜 RAG가 필요할까?

일반 LLM만 쓰면 이런 문제가 있음.

<br>

### 1. 모르는 걸 그럴듯하게 말함 (환각, Hallucination)

학습 데이터에 없거나 확실하지 않은 내용도 **자신 있게** 지어낼 수 있음.

```text
질문: "우리 회사 2025년 휴가 규정 알려줘"

일반 LLM: (규정을 본 적 없지만) 그럴듯한 답을 만들 수 있음
RAG:      (사내 wiki에서 규정 검색 →) 실제 문서 기반으로 답
```

<br>

### 2. 최신 정보를 모름

학습이 끝난 시점 이후의 뉴스, 제품, 사내 문서는 **기본적으로 모름**.

<br>

### 3. 우리만의 자료를 모름

회사 내부 문서, 내 노트, PDF, 매뉴얼 같은 **비공개 자료**는 학습에 안 들어감.

<br>

✅ 정리

> **RAG의 핵심 아이디어 = "모든 걸 외우게 하지 말고, 필요할 때 찾아보라"**

<br>

## RAG 전체 흐름 — 두 단계

RAG는 **준비 단계**(미리)와 **질문 단계**(매번)로 나뉨.

<br>

### 준비 단계 (Indexing — 미리 해두는 작업)

회사 문서 1000페이지가 있다고 가정:

```text
[1] 문서 모으기
    PDF, 웹페이지, 메모, FAQ 등

[2] 작은 조각으로 자르기 (Chunking)
    긴 문서를 "한 입 크기"로 나눔
    예: "휴가 신청은 HR 포털에서..." (한 단락)

[3] 각 조각을 숫자로 바꾸기 (Embedding)
    02번 Embedding에서 배운 것!
    "의미가 비슷한 문장 → 비슷한 숫자 벡터"

[4] 검색용 저장소에 보관 (Vector DB)
    조각 + 숫자(벡터)를 샅샅이 찾을 수 있게 저장
```

**비유:** 도서관 **카드 목록**을 미리 만드는 것  
(책 전체를 외우는 게 아니라, "이 주제는 몇 페이지" 표를 만들어 둠)

<br>

### 질문 단계 (Query — 사용자가 물을 때)

```text
사용자: "연차는 며칠 남았어?"

[1] 질문도 숫자(벡터)로 변환

[2] 저장소에서 "의미가 가장 비슷한" 조각 3~5개 검색
    → "연차는 입사 1년 후 15일 부여..."
    → "잔여 연차는 HR 포털 > 근태 메뉴에서 확인..."

[3] 질문 + 찾은 조각들을 AI에게 함께 전달
    ┌─────────────────────────────────────┐
    │ [참고 자료]                          │
    │ - 연차는 입사 1년 후 15일...          │
    │ - 잔여 연차는 HR 포털에서...          │
    │                                     │
    │ [질문]                               │
    │ 연차는 며칠 남았어?                   │
    └─────────────────────────────────────┘

[4] AI가 참고 자료를 읽고 답 생성 (Generation)
    "HR 포털 근태 메뉴에서 확인할 수 있습니다.
     규정상 입사 1년 후 15일이 부여됩니다."
```

**중요:** AI는 여기서 참고 자료를 **"읽는" 것**이지, 그걸 **새로 외우는 것**은 아님.  
다음 질문 때도 **다시 검색**함.

<br>

## 한 장면으로 보기

```text
                    ┌──────────────────┐
                    │  우리 문서들       │
                    │  (PDF, 위키, FAQ) │
                    └────────┬─────────┘
                             │ 준비 단계
                             ▼
                    ┌──────────────────┐
                    │  작은 조각 +      │
                    │  의미 숫자(벡터)   │  ← Vector DB
                    └────────┬─────────┘
                             │
  사용자 질문 ────────────────┤
  "휴가 규정 알려줘"           │
                             ▼
                    ┌──────────────────┐
                    │  비슷한 조각 검색   │  ← Retrieval
                    └────────┬─────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │ 질문 + 조각 묶어서  │  ← Augmented
                    │ AI에게 전달        │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │  답변 생성         │  ← Generation
                    │  (06번과 동일)     │
                    └──────────────────┘
```

06번에서 배운 **토큰 선택 반복**은 RAG에서도 **그대로** 일어남.  
차이점은 프롬프트에 **참고 자료가 같이 들어간다**는 것.

<br>

## 핵심 구성 요소

| 구성 요소 | 역할 | 일상 비유 |
|-----------|------|-----------|
| **Documents** | 원본 자료 | 책, 파일, 웹페이지 |
| **Chunk (청크)** | 잘린 조각 | 책의 한 단락 |
| **Embedding** | 의미를 숫자로 | "주제 지도" 위의 좌표 (02번) |
| **Vector DB** | 빠른 유사도 검색 | 도서관 검색 시스템 |
| **Retriever** | 찾아오는 역할 | 사서 |
| **LLM** | 글 쓰는 역할 | 자료를 읽고 설명하는 사람 |
| **Prompt** | 지시 + 자료 + 질문 | "이 자료만 보고 답해"라고 적힌 시험지 |

<br>

## 02번 Embedding과의 연결

RAG 검색은 **단어가 똑같지 않아도** 의미가 비슷하면 찾아야 함.

```text
  문서 조각:  "연차 휴가는 입사 1년 후 부여됩니다"
  사용자 질문: "쉬는 날 몇 일이야?"

  단어는 다르지만 → Embedding으로 변환하면 → 숫자 좌표가 가까움 → 검색 성공!
```

**비유:** 도서관에서 "고양이"라고 검색했는데,  
카드에는 "Felis catus"라고 적혀 있어도 **의미가 같으면** 찾아줌.

<br>

✅ 정리

> RAG 검색의 핵심 = **키워드 매칭**이 아니라 **의미 유사도** (Embedding)

<br>

## RAG vs 그냥 LLM vs Fine-tuning

| 방식 | 쉬운 설명 | 장점 | 단점 |
|------|-----------|------|------|
| **그냥 LLM** | 외운 것만으로 답 | 간단, 빠름 | 최신·사내 정보 부정확 |
| **RAG** | 필요할 때 자료 찾아서 답 | 최신·사내 문서 반영 쉬움, 출처 표시 가능 | 검색 품질에 따라 답 품질 좌우 |
| **Fine-tuning** | AI 자체를 다시 가르침 | 말투·도메인에 특화 | 비용·시간 큼, 문서 바뀔 때마다 다시 학습 |

<br>

### 언제 RAG?

* 회사 FAQ, 매뉴얼, 논문, 위키 기반 Q&A
* "우리 자료 기준으로 답해줘"
* 문서가 **자주 바뀜** (RAG는 문서만 갈아끼우면 됨)

<br>

### 언제 RAG 말고 다른 것?

* 단순 잡담, 창작, 일반 상식 → **그냥 LLM**으로 충분
* AI **말투·행동 자체**를 바꾸고 싶음 → **Fine-tuning**
* 검색 + 계산 + 여러 도구를 **스스로 조합** → **AI Agent** (다음 학습 주제)

<br>

## RAG가 잘 되려면?

<br>

### 1. 좋은 자료가 있어야 함

```text
  쓰레기 문서 → 쓰레기 답
  (검색해도 관련 내용이 없으면 AI는 또 지어낼 수 있음)
```

<br>

### 2. 자르는 방법(Chunking)이 중요

```text
  너무 크면 → 핵심만 못 찾음 (한 조각에 너무 많은 내용)
  너무 작으면 → 문맥이 끊김 ("...신청은 HR" / "포털에서...")
```

**비유:** 피자 자르기 — 한 입에 넣을 크기로, 하지만 토핑이 다 보이게

<br>

### 3. 검색이 정확해야 함

질문과 **단어가 같지 않아도** 의미가 비슷하면 찾아야 함 → Embedding이 핵심.

<br>

### 4. "이 자료만 보고 답해"라고 시켜야 함

프롬프트에 이런 규칙을 넣으면 환각이 줄어듦:

```text
  "아래 참고 자료만 사용해서 답하세요.
   자료에 없으면 '모르겠습니다'라고 하세요."
```

<br>

## 자주 나오는 용어

| 용어 | 한 줄 설명 |
|------|------------|
| **Context** | AI에게 같이 넘겨주는 참고 텍스트 |
| **Top-K** | 가장 관련 있는 조각 K개만 가져옴 (예: 상위 3개) |
| **Similarity search** | 의미가 비슷한 조각 찾기 |
| **Hallucination** | 근거 없이 그럴듯하게 지어내기 |
| **Grounding** | 답을 실제 자료에 "뿌리" 내리기 |
| **Citation** | "이 답은 3페이지 규정에서 왔어요"처럼 출처 표시 |
| **Vector DB** | Embedding 숫자를 저장·검색하는 DB |

<br>

## 실생활 예시

<br>

### 예시 1: 회사 챗봇

```text
  자료: 사내 wiki, HR 규정 PDF
  질문: "재택근무 신청 방법?"
  RAG:  wiki에서 '재택근무' 관련 문단 검색 → 그걸 보고 단계별 안내
```

<br>

### 예시 2: 논문 Q&A

```text
  자료: 논문 PDF 50편
  질문: "이 연구에서 사용한 데이터셋은?"
  RAG:  논문 중 'dataset' 관련 단락 검색 → 해당 논문 내용만 인용해 답
```

<br>

### 예시 3: ChatGPT + 파일 업로드

파일을 올리고 "이 문서 요약해줘"라고 하면,  
내부적으로 **비슷한 RAG/컨텍스트 주입** 개념으로 그 파일 내용을 참고함.

<br>

### 예시 4: 대표 RAG AI 서비스

아래 서비스들은 **RAG 방식**(검색 → 자료 붙이기 → 답 생성)을 쓰거나, **기능을 켰을 때** RAG와 비슷하게 동작함.  
(모든 AI 챗봇이 **항상** RAG를 쓰는 건 아님 — 아래 **ChatGPT는 LLM이면서 RAG도 쓰는 거야?** 참고)

<br>

#### 거의 항상 검색(Retrieval)이 붙는 서비스

| 서비스 | RAG가 어떻게 쓰이는지 |
|--------|----------------------|
| **Perplexity** | 질문할 때마다 웹 검색 → 관련 페이지를 찾아 프롬프트에 넣고 답 생성. **출처 링크**를 같이 보여 줌. RAG 중심 Q&A의 대표 사례 |
| **Google Gemini (검색 연동)** | Google 검색 결과를 근거로 답함. 최신 뉴스·가격 같은 **사실·최신 정보** 질문에 RAG(grounding) 사용 |
| **Microsoft Copilot** | Bing 검색 + (기업용) SharePoint·Teams 문서 검색. 회사 내부 자료 기반 답변에 RAG 구조가 많음 |
| **Glean** | Slack, Drive, Jira 등 **사내 문서 전체**를 검색해서 답하는 엔터프라이즈 검색+챗봇. 전형적인 사내 RAG |

<br>

#### 기본은 LLM, 기능 켤 때 RAG

| 서비스 | RAG가 붙는 경우 |
|--------|----------------|
| **ChatGPT** | 웹 검색(Browse), **파일 업로드**, **Custom GPT(지식 파일)** — 일반 채팅만 할 때는 RAG 없음 |
| **Claude** | 웹 검색, **Projects**에 올린 문서 참조 |
| **Notion AI / Slack AI** | 워크스페이스·채널 안의 문서를 검색해 답함 (사내 지식 RAG) |
| **Intercom Fin** 등 고객지원 봇 | FAQ·매뉴얼·티켓 DB 검색 후 답변 (고객센터 RAG) |

```text
[일반 질문]     → LLM만 (학습 지식 + 대화 맥락)
[검색/문서 ON]  → Retrieval → Augmented → Generation  ← RAG
```

<br>

#### 분야별로 많이 쓰이는 RAG

| 분야 | 예 |
|------|-----|
| **회사 챗봇** | HR 규정, 사내 wiki 검색 후 안내 (위 예시 1과 동일) |
| **문서 Q&A** | 논문·계약서·매뉴얼 PDF 업로드 후 질의 (위 예시 2, 3) |
| **법률·의료** | Harvey 등 — 판례·가이드라인 검색 후 답변 (출처 표시가 중요) |
| **직접 만드는 앱** | LangChain, LlamaIndex로 RAG 파이프라인 구축 — 수많은 B2B·사내 도구가 이 방식 |

<br>

✅ 한 줄로

> **대표 사례: Perplexity(웹), Glean·Copilot(기업 문서), ChatGPT·Claude(검색·파일·프로젝트 켤 때).** 많은 챗봇은 항상 RAG가 아니라 **자료가 필요할 때만** 검색 단계를 붙임.

<br>

## RAG의 한계

1. **검색을 못 하면** 답도 틀림 — "못 찾은 것"과 "없는 것" 구분 어려움
2. **자료가 너무 많으면** 어떤 조각을 넣을지 고르기 어려움
3. **복잡한 추론** (여러 문서를 깊게 연결)은 여전히 어려움 → **AI Agent**와 함께 쓰는 경우 많음
4. **답은 맞는데 출처가 틀린** 경우도 있음 — 출처 표시가 있어도 100%는 아님

<br>

## 전체 학습 흐름에서의 위치

```text
  01 Tokenization     → 글을 조각으로
  02 Embedding        → 조각을 숫자로        ← RAG 검색에 사용
  03 Attention        → 문맥 이해
  04 Transformer      → 깊은 이해
  05 Next Token       → 다음 토큰 확률
  06 Inference & Gen  → 실제로 답 생성      ← RAG의 Generation 단계
  07 RAG              → 자료 찾아서 답 생성  ← 여기!
  08 AI Agent         → (다음) 도구·검색을 스스로 조합
```

<br>

## 핵심 정리

1. **Retrieval** — 관련 조각 찾기
2. **Augmented** — 질문에 조각 붙이기
3. **Generation** — 붙인 걸 보고 답 쓰기 (06번과 동일)
4. **Indexing** — 문서를 미리 잘라서 Vector DB에 저장 (준비 단계)
5. **환각 방지** — 외우지 않고, 필요할 때 검색해서 답함

<br>

✅ 한 문장으로

> **RAG = AI에게 "외우지 말고, 필요할 때 찾아 읽고 답하라"고 시키는 방법**

```text
  일반 LLM  :  질문 ──────────────────▶ 답
  RAG       :  질문 ──▶ 자료 검색 ──▶ (질문+자료) ──▶ 답
```

**비유:** 스마트한 도서관(검색) + 글 잘 쓰는 AI(생성)

<br>

## 다음 단계

다음 학습 주제는 **AI Agent**.

* RAG에 **"직접 검색하고, 여러 번 시도하고, 도구를 쓰는"** 능력이 더해진 형태
* 목표 달성할 때까지 스스로 검색·계산·재시도

```text
  RAG     :  "이 자료 찾아서 답해" (1회 검색 → 1회 답)
  Agent   :  "목표 달성할 때까지 스스로 검색·계산·재시도" (여러 단계)
```

RAG가 **"무엇을 참고해서 쓸지"**라면,  
AI Agent는 **"어떻게 스스로 문제를 풀지"**에 가까움.

<br>

## 내가 이해한 내용

RAG는 답하기 전에 관련 문서를 검색해서 프롬프트에 붙이고,  
06번에서 배운 방식(Inference & Generation)으로 답을 만든다.  
모델이 문서를 외우는 게 아니라, 매 질문마다 다시 검색한다.

<br>

## 아직 헷갈리는 부분

아래는 학습 중 질문했던 내용을 다시 정리한 것. (헷갈렸던 지점 → 이해한 버전)

<br>

### RAG의 Generation이랑 06번 Inference & Generation이랑 같은 거야?

**완전히 같은 말은 아님.** RAG의 Generation은 06번과 **같은 답 쓰기 엔진**을 가리키지만, **범위가 다름**.

| 용어 | 범위 |
|------|------|
| **Inference & Generation (06번)** | 질문만 넣고 → 모델 돌리고 → 토큰 고르고 → 반복해서 답 완성하는 **전체 과정** |
| **RAG의 Generation (G)** | RAG 3단계 중 **마지막 단계** — "찾은 자료를 보고 답 쓰기" |

RAG는 그 앞에 두 단계가 더 붙음:

```text
Retrieval   → 관련 자료 검색
Augmented   → 질문 + 자료를 프롬프트에 붙이기
Generation  → 그걸 보고 답 작성  ← 여기가 06번과 겹침
```

```text
일반 LLM (06번):
  질문 ──────────────────────────▶ Inference & Generation ──▶ 답

RAG (07번):
  질문 ──▶ Retrieval ──▶ Augmented ──▶ Generation ──▶ 답
                                      └─ 여기 = 06번과 동일한 엔진
                                         (단, 프롬프트에 자료 포함)
```

**RAG Generation 단계 안에서는?**  
06번과 똑같이 **Inference + Generation이 토큰마다 반복**됨.

```text
[RAG Generation 단계 내부]

질문 + 참고자료 → Transformer → 확률표 (Inference)
                              → 토큰 1개 선택 (Generation)
                              → 붙이고 반복
                              → 답 완성
```

| 정리 | 내용 |
|------|------|
| RAG의 Generation ≠ Inference & Generation 전체 | RAG는 검색·보강이 앞에 있음 |
| RAG의 Generation = 06번과 같은 답 생성 방식 | 토큰 단위 반복 (Autoregressive) |
| RAG Generation 안에서도 | Inference(확률 계산)와 Generation(토큰 선택·붙이기)이 **같이** 일어남 |

**비유:** RAG의 Generation = 도서관에서 책 찾아온 뒤(검색) 시험지에 붙여 넣고(보강) **06번에서 배운 방식으로 답 쓰는 것**. 답 쓰는 방법 자체는 같고, **참고 자료가 프롬프트에 같이 들어간다**는 점만 다름.

<br>

### RAG는 모델이야?

**아님.** RAG는 Transformer 같은 **하나의 신경망 모델**이 아니라, **여러 구성 요소를 엮은 방식(파이프라인)**.

| 구분 | 설명 |
|------|------|
| **LLM (예: GPT)** | 답을 **생성**하는 모델 |
| **임베딩 모델** | 문서·질문을 **숫자(벡터)**로 바꾸는 모델 (검색에 사용) |
| **검색 시스템** | Vector DB, 키워드 검색 등 (모델이 아닐 수도 있음) |
| **RAG** | 위 것들을 **조합한 설계** — "찾아 읽고 나서 답하라"는 방법 |

```text
RAG = 검색(Retrieval) + (선택) 임베딩 모델 + LLM(Generation)
```

**비유:**

| 대상 | 비유 |
|------|------|
| **LLM** | 요리사 (레시피를 외워서 요리) |
| **RAG** | 요리 **전에** 냉장고를 열어 재료를 확인한 뒤 요리하는 **방법** |

요리사(LLM)는 그대로인데, **어떻게 정보를 가져와 쓸지**를 바꾼 것.

<br>

### ChatGPT는 LLM이면서 RAG도 쓰는 거야?

**반은 맞고, 반은 다름.** ChatGPT는 **항상 LLM을 쓰지만**, **매번 RAG를 쓰는 건 아님**. 기능에 따라 달라짐.

| 기능 | LLM | RAG(또는 비슷한 방식) |
|------|-----|----------------------|
| **일반 채팅** | ✅ 항상 | ❌ 없음 — 학습 지식 + 대화 맥락만으로 답 |
| **웹 검색 / Browse** | ✅ | ✅ 웹에서 페이지 검색 → 프롬프트에 넣기 → 답 생성 |
| **파일 업로드** | ✅ | ✅ 문서 내용 검색·참조 → 답 생성 |
| **Custom GPT (지식 파일)** | ✅ | ✅ 미리 올린 문서 검색 → **전형적인 RAG에 가까움** |

```text
[일반 질문]  "파이썬이 뭐야?"
  → LLM만 → 답변

[웹 검색 ON]  "오늘 서울 날씨 어때?"
  → 웹 검색(Retrieval) → 결과 붙이기(Augmented) → LLM 답(Generation)

[PDF 업로드]  "이 문서 3장 요약해줘"
  → 문서에서 관련 부분 찾기 → LLM에 넣기 → 답변
```

**헷갈리기 쉬운 점 — Memory(기억) ≠ RAG**

| | RAG | Memory |
|---|-----|--------|
| **언제** | 질문할 때마다 | 대화 사이에 미리 저장 |
| **무엇** | 외부 자료(문서, 웹) 검색 | 사용자에 대한 정보 기억 |
| **비유** | 매번 도서관에서 책 찾기 | 메모장에 적어 두고 다음에 꺼내 보기 |

> **ChatGPT = 기본적으로 LLM + (필요할 때) 검색 단계를 붙이는 구조**

<br>

### 자주 헷갈리는 질문 정리

**Q1. RAG하면 AI가 문서를 "외우"나?**  
→ **아님.** 매 질문마다 **검색해서 읽고** 답함.  
문서가 바뀌면 저장소만 업데이트하면 됨 (모델 재학습 불필요).

**Q2. RAG와 Fine-tuning 중 뭐가 더 좋나?**  
→ **목적이 다름.**  
RAG = "특정 자료 기반으로 답하기" / Fine-tuning = "AI 자체 성격·말투 바꾸기"  
실무에서는 **둘 다 같이** 쓰기도 함.

**Q3. ChatGPT도 RAG를 쓰나?**  
→ **항상은 아님.** LLM은 항상 쓰고, 웹 검색·파일 업로드·Custom GPT 등에서만 RAG와 비슷한 원리 사용. (위 **ChatGPT는 LLM이면서 RAG도 쓰는 거야?** 참고)

**Q4. RAG는 모델이야?**  
→ **아님.** LLM·임베딩 모델 등을 **조합한 파이프라인/방식**. (위 **RAG는 모델이야?** 참고)

<br>

✅ 한 문장으로 (Q&A 정리)

> **RAG는 모델이 아니라 "찾아 읽고 답하는" 방법이고, ChatGPT는 항상 LLM을 쓰되 RAG는 기능을 켰을 때만 붙는다.**
